# Расширенное техническое описание системы RecipeFinder

## 1. Концепция и назначение
**RecipeFinder** — это интеллектуальная система подбора кулинарных рецептов, ориентированная на минимизацию пищевых отходов и упрощение процесса готовки. Приложение решает задачу: «Что я могу приготовить из того, что уже есть в холодильнике?», используя при этом не только строгие алгоритмы поиска, но и гибкую систему замен вместе с мощностями современных языковых моделей (LLM).

---

## 2. Архитектурный стек и паттерны
Приложение спроектировано с соблюдением принципов **SOLID** и **Clean Architecture**, что позволяет изолировать бизнес-логику от деталей реализации (БД, сеть, UI).

*   **Pattern:** MVVM (Model-View-ViewModel).
*   **DI:** Hilt (Dagger 2) — используется для внедрения зависимостей на всех уровнях приложения.
*   **Async/Reactive:** Kotlin Coroutines & Flow — обеспечивают плавную работу UI и реактивность данных.
*   **UI:** Jetpack Compose — полностью декларативный подход к построению интерфейса.

---

## 3. Слой данных (Data Layer)

### 3.1. Локальное хранилище (Room)
База данных SQLite (через Room) является «источником истины».
*   **Индексация FTS4:** Таблица ингредиентов дублируется виртуальной таблицей FTS (Full-Text Search), что обеспечивает мгновенный поиск среди тысяч продуктов при вводе каждой буквы.
*   **Связи «Многие-ко-многим»:** Реализованы через таблицу-перемычку `RecipeIngredientCrossRef`, связывающую рецепты с их компонентами.
*   **Система маппинга:** Таблица `ingredient_substitutions` хранит связи между базовыми ингредиентами и их аналогами (например, Сыр -> Моцарелла).
*   **AI Cache:** Таблица `ai_cache` хранит результаты работы нейросети (ID рецепта + хэш корзины -> оценка). Это исключает повторную оплату токенов за одни и те же запросы.

### 3.2. Сетевое взаимодействие (OpenRouter)
*   **Retrofit:** Настроен на работу с OpenRouter API.
*   **OkHttp:** Включает интерцептор логирования для отладки сетевых запросов.
*   **Kotlinx Serialization:** Типобезопасная десериализация JSON-ответов от LLM.

## 4. Слой бизнес-логики (Domain Layer)

### 4.1. Алгоритм подбора (Smart Matching)
`GetRecommendedRecipesUseCase` реализует сложную логику расчета `matchScore`:
1.  Загружает все доступные замены в память (Map) для оптимизации скорости.
2.  Для каждого рецепта проверяет наличие ингредиента:
    *   Если ID ингредиента есть в корзине — **Match**.
    *   Если ингредиента нет, но в корзине есть его **заменитель** из маппинга — **Match**.
3.  Результат возвращается в виде `RecipeSearchResult` (обертка над рецептом с рассчитанным весом).

### 4.2. ИИ-оценка (AI Logic)
`GetRecipeAiEvaluationUseCase` отправляет контекстный запрос к модели **Gemma-3**:
*   **Context:** Передаются названия продуктов пользователя, состав рецепта и, что критически важно, **пошаговая инструкция приготовления**.
*   **Prompt Engineering:** Промпт настроен так, чтобы ИИ возвращал строго числовое значение, анализируя кулинарную совместимость продуктов.

---

## 5. Слой представления (Presentation Layer)

### 5.1. Управление состоянием (ViewModel)
`MainViewModel` координирует работу всех UseCase:
*   Поддерживает `MutableStateFlow` для корзины, результатов поиска и списка ИИ-оценок.
*   **Асинхронная загрузка:** ИИ-оценка для первого рецепта (лимит для стабильности) запускается в отдельной корутине, не блокируя отображение списка.

### 5.2. Пользовательский интерфейс (Jetpack Compose)
*   **Динамическая корзина:** Поддерживает FlowLayout для компактного отображения выбранных продуктов.
*   **Визуализация замен:** В карточке рецепта отображается полупрозрачный зеленый блок `Замена: А -> Б`, если алгоритм использовал аналог.
*   **Индикация ИИ:**
    *   **Скелетон:** Пульсирующая анимация (Alpha Transition) во время ожидания ответа от OpenRouter.
    *   **Цветовое кодирование:** Зеленый круг — математический расчет, Синий круг + бейдж "ИИ" — экспертная оценка нейросети.
*   **Экран деталей:** Использует `LaunchedEffect` для асинхронного определения, какие ингредиенты помечены как "имеющиеся" за счет замен.

---

## 6. Безопасность и конфигурация
*   **BuildConfig:** Использование `buildConfigField` позволяет безопасно внедрять API-ключи из `local.properties` в код приложения, не допуская их утечки в репозиторий.
*   **Migration Policy:** Внедрена политика `destructiveMigration`, упрощающая обновление схемы БД в процессе активной разработки.
*   **Permission:** Контроль доступа к INTERNET для работы ИИ-модуля.
